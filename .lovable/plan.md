## הבעיה

באתר בערבית, הוריאציות של מארזים מופיעות בעברית.

### שורש הבעיה

ב־`src/components/inventory/BundleVariationsManager.tsx` (דיאלוג יצירה/עריכה של וריאציית מארז) שדות הקלט **הפוכים**:

- שורה 200–201: התווית "שם הוריאציה (עברית)" → אבל ה־`onChange` כותב ל־`name`, שזו עמודת **הערבית** במסד.
- שורה 203–205: התווית "שם הוריאציה (ערבית)" → אבל ה־`onChange` כותב ל־`name_he`, שזו עמודת **העברית** במסד.

לפי הסכמה (`bundle_variations.name` = ערבית, `bundle_variations.name_he` = עברית) ולפי הצגה באתר (`WebProductPage.tsx` שורה 373: `lang === "he" ? (name_he || name) : name`), כל הוריאציות שנוצרו עד היום נשמרו עם **טקסט עברי בעמודה הערבית** ולכן באתר בערבית הן מופיעות בעברית.

תופעת לוואי: ב־CRM (שורה 160: `{bv.name}`) הרשימה נראית "תקינה" בעברית כי הטקסט שנשמר באמת עברי.

## הפתרון

### 1. תיקון הדיאלוג ב־`BundleVariationsManager.tsx`

החלפת התוויות (או החלפת ה־states) כך שהשדה המוצג כ"ערבית" באמת ייכתב לעמודה `name`, והשדה "עברית" ייכתב ל־`name_he`. הדרך הנקייה: שינוי שמות ה־states ל־`nameAr` / `nameHe`, ושינוי placeholders שיתאימו (ערבית בשדה ערבית, עברית בשדה עברית). שמירת לוגיקת ה־payload כפי שהיא: `{ name: nameAr, name_he: nameHe }`.

נוסיף גם דרישת מילוי לשם הערבי (זה ה־primary), והעברית כאופציונלית — תואם ליתר המערכת.

### 2. תצוגת רשימת הוריאציות ב־CRM (אותו קובץ, שורה 160)

לאחר התיקון, `bv.name` יהיה ערבית. מאחר שה־CRM הוא בעברית, נשנה ל:
```tsx
<span className="font-medium">{(bv as any).name_he || bv.name}</span>
```
כדי שמנהלים יראו את השם בעברית (בדומה לתבנית `name_ar || name` שכבר נהוגה בפרויקט בכיוון ההפוך).

### 3. מיגרציית נתונים קיימים

כל הוריאציות הקיימות שנוצרו דרך הטופס השגוי מכילות עברית ב־`name` ו־NULL/ריק ב־`name_he`. נריץ מיגרציה שמחליפה את העמודות **רק עבור שורות שבהן `name_he` ריק/NULL** (כדי לא לפגוע בשורות שהוזנו נכון אם יש כאלה):

```sql
UPDATE public.bundle_variations
SET name_he = name,
    name   = NULL  -- יוחלף בשלב הבא בידי המנהלים, או יישאר כעברית עד שיתוקן
WHERE COALESCE(name_he, '') = '';
```

⚠️ אבל זה ישאיר את `name` (הערבית) ריק → באתר ערבית יוצג שם ריק.

לכן מציע פתרון בטוח יותר: **להעתיק** את הערך הקיים גם ל־`name_he` ולהשאיר אותו זמנית ב־`name` כדי שהאתר ימשיך להציג משהו, ולסמן בלוג שצריך לערוך:

```sql
UPDATE public.bundle_variations
SET name_he = name
WHERE COALESCE(name_he, '') = '';
```

לאחר המיגרציה, המנהל ייכנס לכל מארז ויעדכן את שדה הערבית לתרגום הנכון. עד אז, האתר בערבית ימשיך להציג את הערך הנוכחי (שזה הפחות גרוע).

### שאלה למשתמש

האם אחרי המיגרציה אתה רוצה גם לרוקן את `name` (הערבית) כדי שיהיה ברור שצריך למלא — או להשאיר את הטקסט העברי שם זמנית עד שתעדכן ידנית במארזים? אני אניח **השארה זמנית** אלא אם תעדיף אחרת.

## קבצים שישתנו

- `src/components/inventory/BundleVariationsManager.tsx` — תיקון מיפוי שדות, placeholders, ותצוגת רשימה.
- מיגרציה חדשה ב־`supabase/migrations/` — העתקת `name → name_he` היכן ש־`name_he` ריק.

## בדיקות אחרי הביצוע

1. פתיחת מארז עם וריאציות באתר ערבית — שמות מוצגים (בעברית כרגע, אבל בערבית מיד אחרי שתעדכן את השדה החדש).
2. אתר עברית — השמות עדיין מופיעים בעברית (מתוך `name_he`).
3. CRM — רשימת וריאציות מציגה את השם בעברית (`name_he || name`).
4. יצירת וריאציה חדשה — הזנת ערבית בשדה הערבית והזנת עברית בשדה העברית, ובדיקה שהן יוצאות נכון בשני האתרים.
